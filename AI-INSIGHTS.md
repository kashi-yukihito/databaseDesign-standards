# AI活用による知見メモ
2008年頃の設計規約を現代（2026年）の技術スタックにアップデートする過程でAIを活用した。
本書はAIを活用した設計方法論および意思決定プロセスを記録したものである。
今後のAIとの協働開発を円滑に行うための手がかりとなることを期待する。

## 1. 役割分担とワークフロー

### 各モデルの役割分担
以下のAIに規約の技術内容・表現をレビューさせ、指摘を取捨選択しながら規約に反映した。(2026年3月時点)
実際に利用方法を試しながら、現時点で最適と感じた役割を「役割定義」として記載した。
執筆時点での技術スタックは妥当と考えているが、AIについては変化が激しいため、適用時期によって用途別の採用AIモデルは検討が必要。
大切なのは複数の役割をAIに分担させると言うコンセプトだと考える。

| AIモデル Ver. | 主な用途 | 役割定義 |
|---|---|---|
| Claude Sonnet 4.6| メインの対話パートナー。規約の添削・議論・技術検証・英訳 | 主対話AI |
| Claude Opus 4.5| 最終レビュー。トークン消費が大きいため利用には注意が必要 | 最終レビューAI |
| OpenAI ChatGPT 5.2 | 技術内容の妥当性検証・セカンドオピニオン | セカンドオピニオンAI |
| Google Gemini 3 Flash | コード補完・基礎英訳。フィードバックループでのコード生成 | コード生成AI |
| DeepL web | 英訳の最終調整（Claude/ChatGPTの翻訳に違和感がある箇所のみ） | 翻訳AI |

* Gemini利用に当たってはIDEとして使用するGoogle Antigravityからのエージェント/チャット機能を使用している。
* Grokは試してみたが、今回の用途にはしっくりこなかったため途中から適用を除外している。
* CopilotはChat GPTのアウトプットに類似していたためChat GPTに一元化した。ただしChatGPTの利用が無償版であった場合、ChatGPTの代替として利用価値はあると考える。
* 英訳についてはClaude Sonnet 4.6が最もコストと品質のバランスが良く気に入っており、DeepL webは最終調整に利用した。(Opus4.5の方がより高品質だがコストが高い)

実務上よくあるミスである。


### 実際の設計プロセス
本規約の制作において、AIを単なるレビューツールではなく**対話的な設計パートナー**として活用した。

具体的には以下のプロセスで規約を育てた：

1. **旧規約のレビュー**: 2008年頃の旧版規約をベースに、現代的な基準との差分をAIと議論。
2. **ブログ記事の規約化**: 個人ブログに蓄積した実務知見（障害事例・運用Tips等）をAIと対話しながら規約条項に昇華。
3. **条項の議論と深化**: 各規約の優先度（☆）・例外条件・理由の妥当性をAIと往復しながら洗練。
4. **AI間相互レビュー**: ひとつのAIで得られたレビュー結果を他のAIにレビューさせることで、抜け漏れや思考の偏り（特定のベストプラクティスへの過剰適合や直前のコンテキスト依存など）が無いかを確認した。
5. **実装検証**: 実際のDDL（`Snippet-CreateTable.sql`など）をAIにレビューさせ、規約との整合性を確認。そのフィードバックが規約改善にも還元された。
6. **英訳**: 日本語版完成後、翻訳方針（DOA用語の統一など）を合意した上でAIが章ごとに翻訳。

## 2. 人間とAIの対立と合意（Design Decisions）

規約策定において、AI（特にGPT系）は一般的・汎用的なベストプラクティスを推奨する傾向があるが、本規約では**DOA（データ中心アプローチ）**を重視しているため、あえてAIの推奨を棄却、あるいは独自ルールを追加した箇所がある。

### 2-1. **自然キー vs サロゲートキー**: 
  * AIはサロゲートキー（UUID等）を優先しがちだが、本規約では「データの同一性」を重視し、論理設計レベルでの自然キーの重要性を明文化した（`L_PKY001`, `L_PKY004`）。
    * 特にChatGPTは頻繁にサロゲートキーを主役にするよう推奨してきた。その理由としては「それがモダンだから」「この規約は時代遅れのレガシー文書と思われるから」といったDOAの本質を無視した理由付けをしてきた。
### 2-2. **SQL規約の実効性**:
  * `S_008` (EXISTS優先) や `S_009` (結果順序への非依存) など、パフォーマンス差が僅かであっても、データ不整合や予測不能な挙動（Flakyな挙動）を防ぐための「安全性」を優先する判断を行った。
  * EXISTS優先、というだけの規約を提示した場合はPostgreSQLの現バージョンでは実装上無意味な点を指摘してくるなど、モダナイズを行う上で有益なアドバイスがAIからは成された。
  * 一方でDOAの重要性を提示すると、EXISTSを採用することが集合論的な意味の明確さに寄与するという整理が得られた。これは、入力として与える前提や観点によってAIの回答の方向性が変化することを示す好例である。
### 2-3. **独自の安全基準**:
  * AIが明示的に指摘しなかった観点（`S_SEC006`: 開発環境でのPII取り扱い禁止 等）は設計者が独自に追加。
  これは元々の規約になかった内容であり、適切な材料や指摘を行わない場合AIが提案しなかったということで興味深い。無から有を生むわけではないという一例として貴重。
### 2-4. **無限ループと無限拡張**:
  * 一度完成した状態で再度レビューを繰り返すと、AIは無限に新しい観点や改善案を提案し続ける。しかも過去の指摘事項との矛盾も無視できない頻度で発生する。このため自分自身の中で善し悪しを判定する軸を持たずにレビューを繰り返すと、どんどん冗長な規約になってしまう。
  開発規約であるのに1章、2章や本文書をレビューした後は「理論的基盤は未整備」といった指摘をされることも多々あった。

## 3. AI時代の規約の役割
AIによるコード生成が一般化した現代において、規約は「人間が読むもの」から**「AI（エージェント）が従うべき制約（Guardrail）」**としての役割が強まる。
本規約をAIにコンテキストとして与えることで、生成されるDDLやクエリの品質を自動的に担保し、レビューコストを大幅に削減することが、このプロジェクトの最終的な狙いでもある。

## 4. 雑感 -迷いと違和感-
AIが次々に提案してくる内容について、自分が詳しくない内容が詳細に描かれていた場合、つい無批判に採用をしたくなることがある。
しかしAIは無謬ではないし、己の知見をアウトプットするという本書の趣旨に沿った場合、自分が理解していない内容を採用することは利用者に対する侮辱であると考える。
2-4.で触れたような無限ループは本当に陥りやすい。
この項については継続してアップデートを行っていく所存である。
