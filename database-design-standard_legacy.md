  データベース/テーブル論理設計標準（PostgreSQL編）

１．はじめに
データベースはビジネスの根幹をなすリソース集合であり、優れたデータベースは優れたビジネスモデルとなります。故にデータベースの設計は重要かつ高度なスキルを要求される事項といえます。
優れたデータベース構造はビジネスルールの変更に強く、またプログラムの保守性も高めることができます。裏をかえせば貧弱なデータベースはそのビジネス全体の失敗を約束するものとなります。
特に近年のビジネス変化の早さにともなうプロダクト変更のスピードに対応するためにも、データ中心アプローチが有効となっているため、データ中心アプローチを意識したデータベースの論理設計を行うことは大変重要な要件となってきています。
こうした優れたデータベースの構築に当たっては、データそのものの適切な分解・整理といったデータの定義に依存しますが、それらの分析結果を適切に利用するためには適切な情報の定義方法という設計における

２．前提条件
本規約を適用するDBMSはPostgreSQL ver8.0以後とします。
PostgreSQL はRDBMSですので、当然ながらRDBMSとしての特徴と制約を踏まえた設計が望まれます。
最近はハードウェアの高性能化も相まって、DBMSをファイルシステムの代替装置として使用する向きもありますが、本規約の中ではそうした使われ方は想定しません。
本規約は汎用的に使用され、将来にわたっての広範な拡張性を担保する必要のあるシステムの開発については考慮しておりません。このためYAGNI原則に基づき、不要なデータ定義を制限する思想で記述されています。ですが、その部分を除外すればそうしたシステムにおけるデータベース設計にも援用可能です。

３．ネーミング標準
３−１．全般（コーディング規約と同一）
N_GEN001: 英語の使用
基本的に２バイト文字での処理には問題があることからASCII文字列範囲での命名が必要であるが、日本語は同音異義語が多いためローマ字表記を行うと意味がわかりにくくなる傾向があるため英単語の組合わせとする。

N_GEN002: 大文字小文字を区別しない

３−２．データベース
N_DBS001: 使用されるドメインが明確になるようなユニークな有意名称をもつこと

３−３．テーブル
N_TBL001: テーブル名称は具体的な意味を表わす名詞表現を使用する
テーブルの内容が具体的にイメージできる名称を設定すること。１語で表現できない場合はアンダースコアで単語をつなぐことができる。この場合最後の単語が名詞になっていればよい。
cakePHPやRuby on Railsなどで設定される複数形にするという規約は適用しない。単複両系を許容するが、同一データベース内では単数形か複数形かは統一されなくてはならない。"orders"のようなテーブル名称を許容するためには複数形のほうがどちらかといえば推奨される。

N_TBL002: テーブル名称にSQLキーワードは使用しない
標準SQLおよびPostgreSQLにおいて、予約語として扱われているキーワードをテーブル名称に使用してはならない。一部のキーワードは指定そのものができないが、将来拡張のためのキーワードは指定できてしまう場合があるため、充分に注意すること。（いわゆる未予約語）
下記リンク内の表C-1.に指示される語句は使用しないこと。
ref) SQLキーワード

N_TBL003: 実表にプリフィックスは使用しない
ビュー及び一時表にプリフィックスを設定する代わりに、実表には原則としてプリフィックスを設定してはならない。

N_TBL004: 一時表はプリフィックス"tmp_"を用いる
一時表には必ずプリフィックス"tmp_"を設定することとする。混乱を避けるためにも"tmp"から始まる文字列を実表名にすることは推奨されない。

N_TBL005: バックアップ目的の履歴テーブルにはサフィックス"_hist"を用いる
トランザクションテーブルの過去履歴のバックアップや、マスターの変更履歴などの履歴テーブルにはサフィックスとして"_hist"を使用する。
例）受注ファイルの履歴テーブル  => order_hist
    商品価格変更履歴 => product_sales_price_hist

N_TBL006: 履歴自体が目的の履歴テーブルにはサフィックス"_log"を用いる
例）ユーザーのアプリケーション操作履歴　=> user_operation_log

N_TBL007: 一時作業用の実表にはプリフィックス"wrk_"とサフィックス"_yymmdd"を用いる
データ調査やデータ一括更新のためなどのために使い捨てる目的で使用する表は（ただし一時表では困る場合）プリフィックス"wrk_"を必ず用いる。この"wrk_"から始まるファイルについてはいつまで削除されては困る、ということを主張するためにサフィックスとして使用期限を設定できる。サフィックスがないテーブルについてはいつでも削除できるという了解の下、運用がなされる。

３−４．カラム
N_COL001: 同一の意味を持つカラム名称はテーブル間で同一とする

N_COL002: カラム名称にSQLキーワードは使用しない
N_TBL002と同等。
N_COL003: 数値型データは単数にする（ISO-11179)

N_COL00x: フラグデータの名称にはサフィックス"_flag"を用いる
真偽値を格納するフラグデータにははフラグであることを明確化する

N_COL00y: 真偽値で表現できない状態データの名称にはサフィックス"_sts"を用いる

N_COL00z: サロゲートキーの名称にはプリフィックス"sk_"を用いる
ビジネス側はあくまでも主キーを別のコード体系で認識しているため、サロゲートキーを使用していることを明示する。

３−５．インデックス
N_IDX001: 主キーはプリフィックス"pkx_" + テーブル名とする
N_IDX002: 主キー以外のインデックスはプリフィックス"idx_" + テーブル名 + フリーとする

３−６．制約
N_CTL001: 制約名はシステム生成による制約名を使用する
制約名は基本的にはシステム生成による制約名を使用してよい。しかし下記のような条件の場合、直接命名をする方が利点が大きいため標準に従い命名をすること。
【制約名指定が望ましい条件】
ａ）外部キー制約
外部キー制約があるテーブルに対してテーブル単位でのリストア等を実施する場合、制約条件が原因で単純なデータINSERTやUPDATE、DELETE処理が行えないことがある。

３−７．ルール

３−８．ビュー
N_VIW001: ビュー名称はプリフィックス "vw_" + 適切な名称とする
ビューのネスト等を防ぐために、ビュー名称にプリフィックスをもうけ、ビューであることを明示する。

３−９．トリガー
N_TRG001: トリガーは作成しない
トリガーを使用すると関連しあったテーブルへのデータ生成等で便利なこともあるが、基本的には隠蔽されているため（もちろん調べられるのだが）プログラマが意図しないでトリガーと同じ操作を作ってしまったり、意図しないデータ操作が発生してデバッグに手間取るなどのリスクがあるため、基本的に使用しない。

３−１０．手続言語

４． 論理設計標準
４－１．データベース設計標準
D_DBS001: template1は変更しない
D_DBS002: サーバロケールはUTF-8
利用する文字コードは特別な理由のない限りUTF-8を使用する。ただし、接続される処理系によって必然性がある場合はこの限りではない。
４－２．テーブル設計標準
D_TBL001: DOAで設計する
D_TBL002: 1テーブルの目的はひとつ
冗長な構造のテーブルはNULL値が多く、見通しが悪くなる。業務要件が複雑でいくつものテーブルを結合しなければならない場合はテーブルを統合するのではなくビューの使用を検討する。
D_TBL003: 主キーは必ず作成する
すべてのテーブルは主キーを必ず持つこととする。（第1正規化は必ず行う）
例外：
ａ）絶対に１レコードしか存在しないテーブル
１レコードしか存在しない場合、ユニークであることは担保されるため必ずしも必要ではない。しかし、混乱を避けるために可能な限り設定することが望ましい。
ｂ）外部より取得したデータを一時的に格納するワークテーブル
頻繁な全件delete/insertが想定される場合は特に設定しないほうが効果的な場合がある。
しかし、もしdeleteが発生しないのであれば連番型のサロゲートキーを適用する。

D_TBL004: サロゲートキー使用の検討
変更に強いデータ構造とするために、設計時にはサロゲートキーを適用できないかどうかを常に検討すること。従来設計の主キー項目はユニークキー制約を設定し、ユニーク性を担保する。
特に経営戦略によって変更される可能性の高い顧客マスターや仕入先マスターのような製品以外のマスターについては検討に値する。設計時には有意の固定長データであっても、業務の拡大による桁不足や規約の変更(ISBNなど)、M&Aによる片寄せなど、いかなる事態も発生しうることは認識しておくべきである。

D_TBL005: サロゲートキーのデータ型は連番(serial)型
ただし、PostgreSQL7.3以後は連番型は一意性を担保しないため、別途主キー制約を必ず設定すること。

D_TBL006: １テーブルに含まれるカラム数は25以下
適切なテーブル分割・正規化の目安として25を越える数のカラムが存在するテーブルについては見直しを検討すること。この25という数値に論理的裏づけは無いので、一義的なふるいとして理解すること。

D_TBL007: 各テーブルのデータライフサイクルを定義すること
すべてのテーブルについて定義をした時点で、そのデータがどのように保持されるのかの方針を明確化しておくこと。（データを一定期間でパージするのか。パージするのならばどのような基準でどのように設定されるのか）

D_TBL008: 賞味期限切れテーブルを残さない
移行処理や、バックアップのためなどの目的で、一時的に使用する実表を作成する場合があるが、これらのテーブルについては目的を達成した時点で、速やかに削除するようにテーブル自体のライフサイクルを明示すること。
ライフサイクルを明示していないテーブルはすべて不要テーブルと判断する。

D_TBL009: 不要なテーブルは作らない
ここでもYAGNI原則。将来的な拡張性について考慮が必要な場合であっても、利用可能性について妥当なもの以外は実装しない。


４－３．カラム設計標準
D_COL001: 小数を含む数値用カラムは原則real/double precisionを使う
Numeric型は演算速度が劣るため、通常使用ではreal型かdouble precision型を要求される桁精度に応じて採用する。ただし、会計用の金額計算や学術計算用の重要データを格納する場合は、Numeric型を採用すること。

D_COL002: Money型は使用しない
PostgreSQL8.3以後ではMoneyは8バイトになったが、8.2以前では4バイトであり、通貨情報を取り扱うには容量不足のため使用しない。また、前項D_COL001が採用されるため自動的にMoney型は使用されないはずである。

D_COL003: 整数値はinteger型で設定する
特別な理由が無い限りsmallinteger / bigint は使用しない。integerは通常使用には充分な精度をもち、かつパフォーマンスに優れるデータ型である。
現今のサーバ資源においてディスク容量及びメモリー容量は個別のカラムにおける数バイトの差異に対して基本的には寛容であるといっていい。

D_COL004: 文字列はVARCHAR型で設定する
明らかに固定長しか格納しない場合を除き、基本的にはVARCHAR型で定義を行う。

D_COL005:固定長文字列はCHARで設定する
電話番号やISBN等の有意のコード体系情報でかつ文字列長が固定されている文字列データについてはCHAR型で定義を行う。またDB容量のサイジングに万全を期すためにはすべてCHARを採用してよい。

D_COL006: フラグデータはboolean型で設定する
サフィックス"_flag"で表現されるデータは必ず真偽の2値のみが格納されるためbooleanを採用すること。

D_COL007: ステータスデータはvarchar型で設定する
真偽だけでは設定できない複数の値をもつサフィックス"_sts"で表現されるデータは、利用される値が数字のみだとしてもすべてvarchar型とする。

D_COL008: ネットワークアドレスはinet型で設定する
cidr型の方が表記の揺らぎに強いが、利用要件によってはinet型の方が汎用性があるため、基本的にinet型を採用することとする。

D_COL00x: 外部キー制約

４－４．インデックス設計標準
D_IDX001: インデックスはB-Tree
特別な用途以外ではB-tree以外は使用する必要はない。ただし複雑な配列を扱う場合はGiSTインデックスの使用を検討することができる。
ハッシュインデックスはPostgreSQLプロジェクト自体から利用を推奨されていない。

D_IDX002: 1テーブルに設定するインデックス数は４つまで
一回の演算に使用できるインデックスはひとつである以上、闇雲にインデックスを設定してもINSERT / UPDATE の効率を落とすだけである。本当に効果のあるインデックスのみを残すために、5個目のインデックスが必要になった場合は不要なインデックスはないか検査の上、置換するのか追加するのかを決定する。

D_IDX003: インデックス設定意図の記述
ユニークキー以外のインデックスについては、どのように利用されることを想定してそのインデックスを設定したかの記録を残す。このことにより、データ構造変化やデータ利用目的変化が発生する際に、既存インデックスの有効性の判断及び新設の必要性判断に寄与することとする。

４－５．ビュー設計標準
D_VIW001: ビューからビューを作らない
深いネストがされたデータ構造は予期せぬ動作や変更の困難さを伴うため、ビューはすべて実表のみから構成されることとする。このためにビュー名称にはプリフィックスを設けている。

D_VIW002: ビューの数はテーブル数の1/4以下
無制限にビューが作成されていくことを防ぐために１データベースあたりに作成してよいビューの数はテーブル数の1/4以下と規定する。この1/4には論理的裏づけは無いが、あまりにビューが増えすぎる場合は非正規化なども検討が必要となることがある。



４－６．トリガー設計標準
D_TRG001: トリガーは作成しない
理由についてはN_TRG001にあるとおり。

６－１．SQL表記方法
S_001:ORDER BY 句での列番号指定は使用しない
SELECT aaaa ,bbbb ,cccc FROM tbl ORDER By 2,1 などは手軽に記述できる方法であるが、SQL92で将来削除されるべき機能として検討されていることと、SELECT項目が追加した場合などに誤った動作を引き起こしたりなどして保守性を低下させるため、永続させる埋め込みコードとしては使用しない。
S_002 : GROUP BY 句での列番号指定は使用しない
S_001と同様

S_005: INSERT文では挿入先カラム名を明確化する
